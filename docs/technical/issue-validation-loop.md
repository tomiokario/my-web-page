# Issue 検証ループ

このドキュメントは、Issue の要求をいきなり複雑な変更へ広げず、まず小さな題材で AI 開発ループを検証するための進め方をまとめたものです。issue 57 の方針である「小さく試す -> 型を固める -> 複雑なものでも試す」を、日常の Issue 対応に落とし込みます。

## 目的

最初の段階では、以下の点を小さな変更で確認します。

- 質問担当が不足仕様を洗い出せるか
- オーケストレータがユーザー窓口を一貫して担えるか
- 実装担当と fresh review の往復が破綻せず回るか
- task state の圧縮と外出しが実用になるか

この段階で運用の型を固めてから、複数ファイル変更や長い仕様整理が必要な Issue へ広げます。

## 最初に選ぶ題材

最初の検証では、次の条件を満たす小さな題材を優先します。

- 変更対象が 1 つか 2 つのファイルに収まる
- 成否をテストまたは画面確認で判定しやすい
- 仕様の不足があっても Issue コメントで補足しやすい
- 失敗しても巻き戻しや再試行の影響が小さい

逆に、以下の題材は最初の 1 周目には向きません。

- 複数ドメインにまたがる大規模変更
- 公開データの大規模更新
- 長い調査や外部依存の整理が前提になる作業
- review 観点が多すぎて完了条件を揃えにくい作業

## 役割分担

### 1. 質問担当

- Issue 本文と関連コメントから不足仕様を抽出する
- 人間への質問案や Issue 草案は親オーケストレータへ返し、質問担当が自律的に Issue を更新しない
- 新しい Issue を作るのは、人間へのインタビューを経て合意した場合だけに限る
- 最小ループで確認したい観点を 3 つから 5 つに絞る

### 2. オーケストレータ

- ユーザーとの対話、ブランチ運用、進捗共有を一貫して担当する
- 変更範囲を小さく保ち、次に検証する観点を明確にする
- task state を短い要約として残し、長い文脈を引きずらない
- ユーザーへの質問、回答の受け取り、最終判断の確認はすべてオーケストレータが行う
- Issue / PR への書き込みは自律的に行わず、必要条件を満たす場合だけ人間確認のうえで行う

### 3. 実装担当

- DocDD と TDD に従って、先に期待動作をテストや確認手順へ落とす
- 小さな差分で red -> green -> refactor を回す
- 変更に必要なドキュメント更新を同じ差分で行う

### 4. Fresh review

- Issue 本文、関連コメント、現在の差分だけを入力として確認する
- 以前の文脈を引き継がず、要求とのズレと検証不足を優先して見る
- 指摘を反映後、別の fresh review で `OK` が出るまで繰り返す

## 窓口ルール

- ユーザー窓口は親オーケストレータだけが持つ
- 質問担当、実装担当、review agent は人間へ直接質問しない
- サブエージェントの提案や質問案は、必ずオーケストレータが整理してから人間へ返す

## Human-in/out/on-the-loop

### Human-in-the-loop

- Question AI が不足仕様を洗い出し、オーケストレータが人間へ質問する段階です
- 新しい Issue を作るのは、人間へのインタビューを経て内容に合意した場合だけです
- サブエージェントは質問案や Issue 草案を返すだけで、Issue を自律更新しません

### Human-out-of-the-loop

- 人間と合意済みの Issue を実装・検証・fresh review で前進させる段階です
- この段階では、仕様の抜け、高リスクな分岐、既存 Issue と人間の意図の不一致が見えない限り、追加質問なしで進めます
- 実装担当は合意済みの完了条件に沿って差分、テスト、ドキュメント更新をまとめて進めます

### Human-on-the-loop

- PR 上で検証結果を共有し、人間が最終的な意図適合とマージ可否を確認する段階です
- 実装中に得た知見は対象 Issue へ自律追記せず、検証元の PR に集約してレビューを受けます
- 人間のレビューで趣旨ずれが見つかった場合だけ、必要な差分修正や補足説明を追加します

## Issue / PR への書き込み条件

Issue や PR への自律的な書き込みはしません。書き込み可能なのは次の場合だけです。

- 質問担当が人間へのインタビューを経て、新しい Issue を作ることに合意した場合
- 実装後のレビュー段階で、既存 Issue だけでは人間の趣旨を十分に表せないと分かり、人間と調整したうえで補足が必要な場合
- 検証結果を集約するために、対象 Issue ではなく検証元の PR へ結果を共有する場合

特に、実装前の方針整理を便利だからという理由だけで対象 Issue に追記しません。

## 最小ループ

1. Issue 本文と関連コメントを読み、目的と完了条件を 1 段落で要約する
2. 最初の題材を 1 つ選び、「今回検証すること」と「今回やらないこと」を分ける
3. 不足仕様がある場合は、オーケストレータが人間への質問として持ち帰る
4. 合意済みの Issue で仕様の抜けがなければ、追加質問なしで期待動作をテスト、あるいは明確な確認手順として先に定義する
5. 小さな差分で実装し、テストまたは確認手順を通す
6. 差分、実行した検証、残課題を短く圧縮して共有する
7. fresh review で要求とのズレ、回帰、ドキュメント更新漏れを確認する
8. 指摘を反映し、別の fresh review で `OK` を確認する
9. 型が固まったら、次の周回で対象を少しだけ複雑にする

## 入力テンプレ

### 質問担当へ渡す最小入力

- 対象 Issue 本文
- 関連コメント
- 人間から追加でもらった制約
- 今回の周回で確認したい論点

期待する出力:

- 不足仕様の一覧
- 人間に聞くべき質問案
- 今回やること / やらないこと

### 実装担当へ渡す最小入力

- 合意済みの仕様
- 対象ファイルと制約
- 期待するテストまたは確認手順
- 完了条件

期待する出力:

- 実装差分
- 実行したテスト
- 残課題

### Fresh review へ渡す最小入力

- 対象 Issue 本文
- 関連コメント
- 現在の差分

期待する出力:

- findings
- open questions / assumptions
- `OK` または残余リスク

## Review 結果フォーマット

review agent の返答は、最低限次の 3 つに揃えます。

- findings: 優先度つきの指摘
- open questions / assumptions: 前提や未確定点
- status: `OK` または残余リスク

指摘が出た場合は、オーケストレータが実装担当へ戻し、別の fresh reviewer に再確認させます。

## 各周回で残すべき最小情報

長い会話をそのまま持ち運ばず、次の 4 点だけ残します。

- 目的: 今回の周回で何を検証したか
- 変更: どのファイルをどう変えたか
- 検証: 何を実行して、何が通ったか
- 残課題: 次の周回で初めて扱う論点は何か

必要に応じて、以下も task state に追加します。

- 現在の停止条件
- 次に呼ぶエージェント種別
- PR へ集約すべき検証結果

## 停止条件と再試行条件

停止して人間へ返す条件:

- 仕様の分岐に人間の意図確認が必要
- 既存 Issue が人間の趣旨を表せていない可能性が高い
- fresh review で高優先度の懸念が解消できない

再試行する条件:

- review で具体的な修正可能指摘が出た
- テスト失敗や差分不足がローカルで解消可能
- 追加の fresh review で `OK` を確認できる見込みがある

## 複雑な対象へ広げる条件

以下が揃ったら、次の段階としてより複雑な Issue に広げます。

- 不足仕様の洗い出しが Issue コメント運用で破綻しない
- 実装と review の往復が 1 周以上安定して回った
- task state の圧縮で次の担当が状況を復元できた
- 変更後の検証方法がチーム内で再利用できる形になった

次の段階では、以下を 1 つずつ追加で検証します。

- 長い仕様整理が必要なケース
- 複数ファイルにまたがる実装
- 複数観点レビューの並列運用
- 文脈膨張を抑えた継続運用
