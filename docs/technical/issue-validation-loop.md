# Issue 検証ループ

このドキュメントは、Issue の要求をいきなり複雑な変更へ広げず、まず小さな題材で AI 開発ループを検証するための進め方をまとめたものです。「小さく試す -> 型を固める -> 複雑なものでも試す」という方針を、日常の Issue 対応に落とし込みます。

この運用は issue 57 で初めて整理しましたが、運用名そのものは issue 番号ではなく内容ベースで扱います。

## 目的

最初の段階では、以下の点を小さな変更で確認します。

- 質問担当が不足仕様を洗い出せるか
- オーケストレータがユーザー窓口を一貫して担えるか
- 実装担当と fresh review の往復が破綻せず回るか
- 一次情報を保持した intent review が最終確認前の趣旨ずれを拾えるか
- task state の圧縮と外出しが実用になるか

この段階で運用の型を固めてから、複数ファイル変更や長い仕様整理が必要な Issue へ広げます。

## 最初に選ぶ題材

最初の検証では、次の条件を満たす小さな題材を優先します。

- 変更対象が 1 つか 2 つのファイルに収まる
- 成否をテストまたは画面確認で判定しやすい
- 仕様の不足があっても Issue コメントで補足しやすい
- 失敗しても巻き戻しや再試行の影響が小さい

逆に、以下の題材は最初の 1 周目には向きません。

- 複数ドメインにまたがる大規模変更
- 公開データの大規模更新
- 長い調査や外部依存の整理が前提になる作業
- review 観点が多すぎて完了条件を揃えにくい作業

## 役割分担

この repo では、役割分担を文書上の概念だけで終わらせず、再利用用の定義として `.codex/agents/*.toml` に置きます。現時点ではこれらの TOML を自動読込する前提ではなく、親オーケストレータが role / model / instructions / 出力形式を実際の subagent 起動設定へ写し、役割ごとの文脈分離を再現します。

合意済み Issue を「対応してほしい」と依頼された場合は、原則としてこの既定フローを一通り回します。追加の subagent 起動や role 分離が一般制約・上位指示・実行環境の都合でできないときだけ、実装に入る前に何が衝突しているか、どの許可や解除が必要かを人間へ先に伝えます。

### 1. 質問担当

- Issue 本文と関連コメントから不足仕様を抽出する
- 人間への質問案や Issue 草案は親オーケストレータへ返し、質問担当が自律的に Issue を更新しない
- 新しい Issue を作るときは、質問担当が質問案や Issue 草案を親オーケストレータへ返し、親オーケストレータが人間と合意した場合だけに限る
- 最小ループで確認したい観点を 3 つから 5 つに絞る

### 2. オーケストレータ

- ユーザーとの対話、ブランチ運用、進捗共有を一貫して担当する
- 変更範囲を小さく保ち、次に検証する観点を明確にする
- task state を短い要約として残し、長い文脈を引きずらない
- 保持する state は、仕様、受け入れ条件、review 結果、一次情報 handoff 参照、次の依頼先に絞る
- implementation agent 起動後は、原則として自分でコードを読み進めて実装都合を抱え込まず、実装詳細が必要になったときだけ implementation agent から最小 handoff を受け取る
- ユーザーへの質問、回答の受け取り、最終判断の確認はすべてオーケストレータが行う
- Issue / PR への書き込みは自律的に行わず、必要条件を満たす場合だけ人間確認のうえで行う
- 人間向けコメントでは内部の role 名や進行用語を使わず、変更内容、確認結果、残課題だけを自然文で共有する

### 3. 実装担当

- DocDD と TDD に従って、先に期待動作をテストや確認手順へ落とす
- 小さな差分で red -> green -> refactor を回す
- 変更に必要なドキュメント更新を同じ差分で行う
- 実装詳細の共有が必要になったときは、親オーケストレータへコード全体の再探索をさせる代わりに、対象ファイル、判断理由、残課題を短い handoff として返す

### 4. Fresh review

- 合意済み Issue の本文、関連コメント、現在の実装差分だけを入力として確認する
- issue 草案や質問フェーズは質問担当で詰め、fresh review の対象にしない
- 以前の文脈を引き継がず、要求とのズレと検証不足を優先して見る
- 指摘を反映後、別の fresh review で `OK` が出るまで繰り返す

### 5. Intent review

- 人間との対話で得た一次情報を保持したまま、今回の実装が趣旨に沿っているかを確認する
- fresh review の `OK` を前提に、要約前の意図ずれや優先順位の取り違えがないかを見る
- intent review 担当は人間窓口ではなく、親オーケストレータが保持する一次情報 handoff を受け取って評価する
- 人間へ直接質問せず、親オーケストレータへ補足説明や追加修正の必要性を返す

### オーケストレータが保持する最小 state

親オーケストレータが常時持つ情報は、次の最小集合に絞ります。

- 対象 Issue / PR と今回の目的
- 合意済み仕様と受け入れ条件
- 今回やること / 今回やらないこと
- 最新の review 結果と未解決事項
- 一次情報 handoff の参照先と `handoff_id`
- implementation agent から受け取った最新 handoff の参照先
- 現在の停止条件、次に依頼する role、対外共有が必要な検証結果

逆に、次は常時保持しません。

- 実装都合の詳細な reasoning
- コードの読み解きメモそのもの
- 人間向けコメントの草稿に内部 role 名や進行用語を混ぜたもの

実装都合の理解が必要になった場合は、親オーケストレータが自分で掘りに行くのではなく、implementation agent に最小 handoff を返させて参照します。

### implementation handoff の最小項目

親オーケストレータが implementation agent へ渡す入力、または implementation agent から返してもらう要約は、少なくとも次を含めます。

- 対象 Issue と今回の目的
- 合意済み仕様と受け入れ条件
- 対象ファイル、非対象、避ける変更
- 必要なテストまたは確認手順
- 既知の findings、未解決事項、関連する制約
- 人間との会話から優先度に影響する引用や決定ログがある場合はその参照先

この handoff には、原則として次を入れません。

- 長い会話ログの丸ごと転送
- 親オーケストレータが引き取る前提の code-level reasoning
- 人間向けコメントの内部用語入り草稿

### 一次情報 handoff の保持

intent review 用の一次情報 handoff は、親オーケストレータが task state とは別の独立ブロックとして保持します。保存先は親オーケストレータ専用の独立メモとし、task state にはその参照先だけを残します。少なくとも次を含めます。

- handoff_id
- 作成者
- 作成日時
- 対象 Issue または PR
- 元になった人間との会話の所在
- 参照元が分かる引用
- 合意した決定ログ

最小テンプレート例:

```text
一次情報 handoff
- handoff_id: intent-20260420-issue57-pr61
- 作成者: parent orchestrator
- 作成日時: 2026-04-20 10:30 JST
- 対象: issue #57 / PR #61
- 元会話の所在: current thread, user message at 2026-04-20 09:55 JST
- 引用:
  - 「fresh review とは別に一次情報を保持した相談AI相当の役割を repo の運用定義へ入れること」
- 決定ログ:
  - intent review は fresh review の `OK` 後に Human-on-the-loop 前で実施する
```

## 窓口ルール

- ユーザー窓口は親オーケストレータだけが持つ
- 質問担当、実装担当、review agent は人間へ直接質問しない
- サブエージェントの提案や質問案は、必ずオーケストレータが整理してから人間へ返す

## Human-in/out/on-the-loop

### Human-in-the-loop

- 質問担当が不足仕様を洗い出し、オーケストレータが人間へ質問する段階です
- 新しい Issue を作るのは、質問担当が質問案や Issue 草案を親オーケストレータへ返し、親オーケストレータが人間と内容合意した場合だけです
- サブエージェントは質問案や Issue 草案を返すだけで、Issue を自律更新しません
- issue 草案や質問フェーズはこの段階で扱い、fresh review には回しません

### Human-out-of-the-loop

- 人間と合意済みの Issue を実装・検証・fresh review で前進させる段階です
- この段階では、仕様の抜け、高リスクな分岐、既存 Issue と人間の意図の不一致が見えない限り、追加質問なしで進めます
- 実装担当は合意済みの完了条件に沿って差分、テスト、ドキュメント更新をまとめて進めます

### Human-on-the-loop

- PR 上で検証結果を共有し、人間が最終的な意図適合とマージ可否を確認する段階です
- この前段で、fresh review に加えて intent review が一次情報ベースの趣旨適合確認を行います
- 実装中に得た知見は対象 Issue へ自律追記せず、検証元の PR に集約してレビューを受けます
- 人間のレビューで趣旨ずれが見つかった場合だけ、必要な差分修正や補足説明を追加します

### 上位制約との衝突時

- subagent 起動や review の一部が上位制約で止まる場合は、実装着手前に停止します
- 何が衝突しているか、どの権限・許可・解除が足りないかを具体的に整理して人間へ伝えます
- その時点でできている検証と、未実施の review や確認も合わせて共有します

## Issue / PR への書き込み条件

Issue や PR への自律的な書き込みはしません。書き込み可能なのは次の場合だけです。

- 質問担当が質問案や Issue 草案を親オーケストレータへ返し、その内容で親オーケストレータが人間と合意して新しい Issue を作る場合
- 実装後のレビュー段階で、既存 Issue だけでは人間の趣旨を十分に表せないと分かり、人間と調整したうえで補足が必要な場合
- 検証結果を集約するために、対象 Issue ではなく検証元の PR へ結果を共有する場合

特に、実装前の方針整理を便利だからという理由だけで対象 Issue に追記しません。

## 人間向けコメントの書き方

PR / Issue コメントやレビュー返信では、内部運用ではなく読み手が必要とする事実だけを共有します。基本は次の 3 点です。

- 何を直したか
- 何を確認したか
- 何が残っているか

避ける表現:

- `implementation agent`
- `fresh review`
- `intent review`
- `検証ループを 1 周完了しました`
- `handoff を更新しました`

言い換え例:

- `○○の条件分岐を修正しました`
- `△△のテストを追加し、既存テストと合わせて通過を確認しました`
- `残っている確認点は □□ です`

内部 role 名や進行概念が必要なのは、repo 内の運用手順書を更新するときだけです。人間向けコメントでは自然文へ置き換えます。

## role 混線が起きたときの立て直し手順

親オーケストレータが実装 reasoning を抱え込み始めた、または対外コメントへ内部用語を書きかけた場合は、そのまま進めません。次の順で立て直します。

1. どの情報が「仕様・受け入れ条件・review 結果」なのか、どの情報が「実装詳細」なのかを切り分ける
2. 親オーケストレータの task state から、実装詳細の読み解きメモや内部用語入り草稿を外す
3. 必要なら implementation agent へ戻し、対象ファイル、判断理由、残課題だけの最小 handoff を作り直す
4. 人間向けコメントは、「何を直したか」「何を確認したか」「何が残っているか」の自然文に書き換える
5. 混線の影響で review 入力が濁った場合は、fresh review からやり直し、必要なら intent review も取り直す

## 最小ループ

1. Issue 本文と関連コメントを読み、目的と完了条件を 1 段落で要約する
2. 最初の題材を 1 つ選び、「今回検証すること」と「今回やらないこと」を分ける
3. 不足仕様がある場合は、オーケストレータが人間への質問として持ち帰る
4. 合意済みの Issue で仕様の抜けがなければ、追加質問なしで期待動作をテスト、あるいは明確な確認手順として先に定義する
5. 小さな差分で実装し、テストまたは確認手順を通す
6. 差分、実行した検証、残課題を短く圧縮して共有する
7. fresh review で、合意済み Issue の実装差分に対する要求とのズレ、回帰、ドキュメント更新漏れを確認する
8. 指摘を反映し、別の fresh review で `OK` を確認する
9. intent review で、一次情報ベースの趣旨適合確認と Human-on-the-loop へ渡す補足要否を確認する
10. 型が固まったら、次の周回で対象を少しだけ複雑にする

## 完了報告の必須項目

Issue の対応を終えて人間へ共有するときは、少なくとも次を明記します。

- 何を直したか
- 何を確認したか
- fresh review を実施したか
- intent review を実施したか
- どちらか未実施なら、その理由
- 進行を止めているブロッカーがあれば、その内容

## 入力テンプレ

### 質問担当へ渡す最小入力

- 対象 Issue 本文
- 関連コメント
- 人間から追加でもらった制約
- 今回の周回で確認したい論点

期待する出力:

- 不足仕様の一覧
- 人間に聞くべき質問案
- 今回やること / やらないこと

### 実装担当へ渡す最小入力

- `implementation handoff の最小項目` に挙げた情報
- 既知の review 指摘を渡す場合は、修正対象に直結するものだけ
- 長い会話ログを渡す場合は、引用元や決定ログが必要な最小範囲だけ

期待する出力:

- 実装差分
- 実行したテスト
- 残課題

### Fresh review へ渡す最小入力

- 対象 Issue 本文
- 関連コメント
- 現在の差分

期待する出力:

- findings
- open questions / assumptions
- `OK` または残余リスク

### Intent review へ渡す最小入力

- fresh review が `OK` になったことを示す確認結果
- 人間との対話で得た一次情報のメモ
- 上記メモの参照元が分かる引用または決定ログ
- fresh review 後の現在の差分

上の 4 点は必須です。いずれかが欠ける場合、intent review は `OK` を出さず、親オーケストレータへ差し戻します。

期待する出力:

- findings
- open questions / assumptions
- `OK` または残余リスク

## Review 結果フォーマット

review agent の返答は、最低限次の 3 つに揃えます。

- findings: 優先度つきの指摘
- open questions / assumptions: 前提や未確定点
- status: `OK` または残余リスク

指摘が出た場合は、オーケストレータが必要に応じて実装担当または一次情報 handoff の更新へ戻し、fresh reviewer または intent reviewer の別個体に再確認させます。

## 各周回で残すべき最小情報

長い会話をそのまま持ち運ばず、次の項目を残します。

- 目的: 今回の周回で何を検証したか
- 変更: どのファイルをどう変えたか
- 検証: 何を実行して、何が通ったか
- 残課題: 次の周回で初めて扱う論点は何か
- 趣旨確認: Human-on-the-loop 前に intent review が返した status と参照先は何か
- 一次情報 handoff 参照: 要点だけに圧縮しない会話メモ、引用、決定ログの参照先と `handoff_id`

別メモに置く運用メタ情報:

- 現在の停止条件
- 次に呼ぶエージェント種別
- PR へ集約すべき検証結果

## 停止条件と再試行条件

停止して人間へ返す条件:

- 仕様の分岐に人間の意図確認が必要
- 既存 Issue が人間の趣旨を表せていない可能性が高い
- fresh review で高優先度の懸念が解消できない
- intent review で一次情報との不一致、または Human-on-the-loop 前に補足が必要という判断が出た

再試行する条件:

- review で具体的な修正可能指摘が出た
- テスト失敗や差分不足がローカルで解消可能
- 追加の fresh review で `OK` を確認できる見込みがある

## 複雑な対象へ広げる条件

以下が揃ったら、次の段階としてより複雑な Issue に広げます。

- 不足仕様の洗い出しが Issue コメント運用で破綻しない
- 実装と review の往復が 1 周以上安定して回った
- task state の圧縮で次の担当が状況を復元できた
- 変更後の検証方法がチーム内で再利用できる形になった

次の段階では、以下を 1 つずつ追加で検証します。

- 長い仕様整理が必要なケース
- 複数ファイルにまたがる実装
- 複数観点レビューの並列運用
- 文脈膨張を抑えた継続運用
